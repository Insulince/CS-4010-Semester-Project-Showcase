This is my item "database". It is simply a JSON file that I write JSON objects too using GSON, however it works exactly like a database for these purposes. The real difference it that you have to define your own functions for CRUD functionality, but that's not too bas. The objects you see here are JSON representations of the Item class. The way I initially populated this file is by creating a bunch of Item objects and running them through Gson's serializer to turn them into these JSON objects. Anytime I need to fetch one as a Java object again, I run the JSON of that object through Gson's deserializer and it's as if the object never left. The reason I had to make a previous and next identifier instead of just pointing to the previous and next objects themselves, is because it becomes impossible to serialize that object into JSON if it works that way. What happens is Gson starts serializing all the attributes of the current object, and once it reaches an attribute pointing to an object of the same type (Item) it attempts to deserialize that by going a level deeper. Then it finds that objects attribute of the same type and goes deeper yet. It does this on repeat until a stack overflow is encountered and the program terminates leaving the database empty. So my solution was to mimic "pointers" by listing the next and previous item's identifiers and not the item itself. When it comes time to fetch the next or previous item, we simply fetch the item from the database matching that identifier and use it.
